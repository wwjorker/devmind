# DevMind 面试讲解稿

这份文档用于帮助你在 Java 后端面试中讲清楚 DevMind。重点不是背项目功能，而是讲清楚：为什么做、怎么设计、解决了什么问题、还有哪些不足。

## 30 秒项目介绍

DevMind 是一个面向个人开发学习材料的 AI 知识库系统。用户可以把八股笔记、项目复盘、Bug 复盘等内容保存为知识文档，系统会自动生成文本分块，然后在提问时先检索相关分块，再构造 Prompt 调用大模型，最后返回带引用来源的回答。

我没有把它做成简单的“调用 AI API 返回答案”的 Demo，而是把 AI 问答接进了一个完整的 Java 后端系统里，包括 JWT 登录、用户数据隔离、文档管理、分块检索、Prompt 构建、LLM Provider 抽象、调用日志、Token 统计、失败兜底、Bad Case 反馈和评估统计。

## 一句话亮点

DevMind 的价值在于：它把 RAG 做成了一个可解释、可观测、可迭代的后端业务链路，而不是薄薄一层 AI API 包装。

## 核心链路

```text
用户登录
-> 创建知识文档
-> 文档自动切分为 chunks
-> 用户提问
-> 提取检索关键词
-> 检索相关 chunks 并计算分数
-> 构建带上下文的 Prompt
-> 路由到 Mock 或 DeepSeek Provider
-> 返回答案和引用
-> 记录 ask log、耗时、token、chunk ids
-> 用户标记 helpful / bad case
-> 汇总评估数据
```

面试时可以强调：这条链路覆盖了后端项目里常见的认证、数据库设计、事务、检索、外部服务调用、异常处理、可观测性和质量反馈。

## 你可以重点讲的设计

### 1. 用户隔离

文档、分块、AI 提问日志、反馈记录都带 `user_id`。接口层从 JWT 中解析当前用户，服务层查询时按用户过滤，避免用户 A 看到用户 B 的数据。

可讲点：这比单纯 Demo 更接近真实系统，因为真实业务必须考虑数据归属和权限边界。

### 2. 文档软删除

删除文档不是物理删除，而是把状态改为归档。这样可以保留历史数据，降低误删风险，也方便以后做恢复、审计或版本记录。

可讲点：软删除是很多企业项目里的常见设计，适合重要数据。

### 3. 文档分块

RAG 不能直接把整篇文档都塞给模型，因为上下文长度和成本有限。项目里创建或更新文档后，会自动生成 chunk，后续检索和引用都基于 chunk。

可讲点：分块让检索粒度更细，也让回答能引用具体来源。

### 4. 可解释检索分数

当前版本不是向量检索，而是关键词检索，并按命中位置计算分数，例如内容、标题、标签命中的权重不同。前端详情页能看到每个 chunk 的 score。

可讲点：虽然这不是最终形态，但它可解释、易调试，适合作为 v1。后续可以升级为向量检索、混合检索和 rerank。

### 5. 无上下文兜底

如果检索不到相关 chunk，系统不会硬让模型乱答，而是返回固定提示：知识库没有足够信息。

可讲点：这是为了降低幻觉。AI 系统不是每次都应该回答，知道什么时候不回答也很重要。

### 6. LLM Provider 抽象

项目里有 `LlmClient` 接口，具体实现包括 Mock 和 DeepSeek。业务服务不直接依赖某一个模型接口，而是通过 Router 选择 Provider。

可讲点：这样方便本地测试、切换模型、控制成本，也避免业务代码和某个厂商 API 强绑定。

### 7. 调用日志和 Token 统计

每次 AI Ask 都记录问题、检索关键词、Prompt Preview、Provider、耗时、Token、召回 chunk ids 和状态。前端可以从日志恢复历史回答和检索上下文。

可讲点：这体现了可观测性。AI 应用不能只关心“答案看起来对不对”，还要能追踪它基于什么上下文回答、花了多少成本、失败在哪里。

### 8. Bad Case 反馈闭环

用户可以把回答标记为 helpful 或 bad case，并填写原因和期望答案。评估页会统计反馈数量、bad case 比例和最近问题。

可讲点：RAG 的效果不是一次写完就结束，要通过 bad case 反推分块、检索、Prompt 和数据质量问题。

### 9. Redis 用途

项目使用 Redis 做 JWT Logout 黑名单。JWT 本身是无状态的，服务端不保存会话，所以退出登录后如果不处理，旧 Token 在过期前仍可能可用。项目把退出的 Token 放入 Redis，并设置剩余有效期 TTL，认证过滤器会检查黑名单。

可讲点：这是一个真实后端问题，不是为了简历硬塞 Redis。

### 10. Flyway 管理数据库迁移

数据库表结构由 Flyway migration 管理，应用启动时自动检查并执行未应用的迁移。

可讲点：这比手动复制 SQL 更工程化，适合团队协作和环境迁移。

## 常见面试追问

### 为什么用 Java 做 AI 项目，而不是 Python？

Python 更适合算法实验和模型训练，但这个项目的定位是 Java 后端岗位作品集。企业里大量业务系统仍然是 Java / Spring Boot 技术栈。DevMind 展示的是如何把 AI 能力接入一个完整后端系统：认证、数据库、事务、日志、缓存、接口设计、外部 API 调用和工程化。

### 现在为什么没直接做向量数据库？

第一版先做可跑通、可解释的 RAG 链路。关键词检索虽然简单，但方便调试，可以清楚看到命中了哪些词、为什么召回这个 chunk。后续升级方向是 embedding、向量检索、关键词和向量混合检索、rerank。

### 怎么减少大模型幻觉？

主要有三层：

1. 先检索知识库，只把相关 chunk 放进 Prompt。
2. Prompt 明确要求只基于提供的上下文回答，并引用 chunk id。
3. 检索为空时不调用模型，直接返回知识库信息不足的兜底回答。

### Prompt Preview 为什么要保存？

因为线上排查时只看最终答案不够。Prompt Preview 能看到当时给模型的完整输入，包括问题、规则和召回上下文。这样可以判断问题出在数据、检索、Prompt，还是模型生成。

### Ask Log 有什么价值？

Ask Log 是 AI 应用的可观测性基础。它记录每次请求的输入、召回、Provider、耗时、Token 和结果状态。后续可以基于它做成本统计、慢请求分析、Bad Case 复盘和质量评估。

### 如果 DeepSeek 调用失败怎么办？

服务层会捕获 Provider 调用异常，记录失败日志，然后返回统一异常。这样失败不是“黑盒”，后续能看到失败发生在哪个问题、哪个 Provider、耗时多少、是否已经召回上下文。

### 这个项目还有哪些不足？

可以诚实说：

- 当前检索还是关键词 v1，不是最终语义检索。
- 还没有文件上传和 PDF 解析，只支持文本输入。
- 还没有流式输出。
- 评估是轻量反馈统计，还不是带标准答案的离线评测集。
- 后续会做向量检索、混合检索、rerank、文件导入和更完整的评估集。

这不是减分项。能清楚讲出边界和下一步，反而说明你真的理解系统。

## 演示时的主线

1. 创建一条 Redis 缓存穿透笔记。
2. 提问：`How should I explain Redis cache penetration in an interview?`
3. 展示回答里的引用、Token、耗时和 chunk ids。
4. 打开 Ask Log Detail，看 Prompt Preview 和 Retrieved Chunks。
5. 提问一个知识库没有的问题，例如 Kubernetes Pod Eviction。
6. 展示无上下文兜底，说明系统不会硬编答案。
7. 保存一个 bad case，进入 Evaluation 页面看反馈闭环。

## 不要过度包装

简历和面试里不要说它是生产级 AI Agent，也不要说已经实现完整向量 RAG。更稳的说法是：

DevMind 是一个面向 Java 后端岗位的 AI 知识库项目，第一版重点打通 RAG 应用的工程链路，并加入可观测性、失败兜底和反馈评估。后续可以继续升级检索质量和文档导入能力。
