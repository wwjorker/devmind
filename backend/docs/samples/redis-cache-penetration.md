# Redis 缓存三兄弟：穿透、击穿、雪崩

## 缓存穿透（Cache Penetration）

**现象**：大量请求查询**根本不存在的数据**（如 id = -1），缓存里没有、数据库里也没有，每次都直接打到数据库。恶意攻击常用这招压垮 DB。

**解决方案**：
- **缓存空值**：查不到时也缓存一个空值，设较短 TTL（如 2 分钟），避免同一个不存在的 key 反复打 DB。
- **布隆过滤器**：请求先过布隆过滤器，判断 key 是否可能存在，不存在直接拒绝。
- **参数校验 / 接口限流**：入口拦掉明显非法的参数。

## 缓存击穿（Cache Breakdown）

**现象**：某个**热点 key 恰好过期的瞬间**，大量并发请求同时发现缓存没了，一起去重建、一起打 DB。是"单个热点 key"的问题。

**解决方案**：
- **互斥锁**：只允许一个线程去查 DB 重建缓存，其余等待/重试（Redis `SETNX` 做分布式锁）。
- **逻辑过期**：缓存不设物理过期，存一个逻辑过期时间；过期后异步重建，旧值先顶着。
- 热点 key 不过期 / 提前预热。

## 缓存雪崩（Cache Avalanche）

**现象**：**大量 key 在同一时刻集中过期**，或 **Redis 整个宕机**，请求瞬间全部打到 DB，DB 被压垮。是"大面积"的问题。

**解决方案**：
- **过期时间加随机值**：避免大量 key 同一秒过期。
- **Redis 高可用**：主从 + 哨兵 / 集群，避免单点。
- **限流降级 + 多级缓存**：DB 前加限流；本地缓存（Caffeine）+ Redis 二级缓存。

## 一句话区分

| | 数据存在吗 | 范围 | 核心 |
|---|---|---|---|
| 穿透 | 不存在 | 持续 | 查不存在的数据 |
| 击穿 | 存在 | 单个热点 key | 热点 key 过期瞬间 |
| 雪崩 | 存在 | 大面积 | 大量 key 同时失效 |

## 延伸

- 布隆过滤器：位数组 + 多个哈希函数；**有误判（说存在不一定存在，说不存在一定不存在）、不支持删除**。
- 缓存空值的缺点：占内存、可能和真实数据短暂不一致（数据后来被创建了）。
