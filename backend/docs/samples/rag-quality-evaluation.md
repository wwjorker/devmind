# RAG 检索质量评估：Hit@K 与 MRR

## 为什么要评估

AI 问答很容易停在"能答就行"的 demo 阶段：能出结果，但不知道质量是否稳定、换了检索策略到底是变好还是变差。要让优化**可衡量**，就得有指标和一批标准问题。

RAG 的质量分两层，**先评检索、再评生成**：检索没召回对的资料，生成再强也是空中楼阁。下面聚焦检索层。

## 核心检索指标

前提是有一批**标注数据**：每个问题人工标好"哪些文档 / 片段才算相关"（gold label）。

- **Hit@K（命中率）**：Top-K 结果里**只要有一个**命中相关文档就算 1，否则 0；对所有问题取平均。衡量"答案有没有进前 K"。
- **MRR（Mean Reciprocal Rank，平均倒数排名）**：取**第一个**命中相关文档的排名，倒数即 `1/rank`（第 1 位=1，第 2 位=0.5，第 3 位≈0.33），对所有问题取平均。衡量"相关结果排得够不够靠前"。
- **Recall@K**：Top-K 命中的相关文档数 / 全部相关文档数，衡量"召回全不全"。
- **nDCG@K**：考虑相关性分级和位置的折损，比 Hit/MRR 更精细。

**Hit@K vs MRR 的区别**：Hit@K 只问"在不在前 K"，不管第几名；MRR 在乎"排第几"。一个策略可能 Hit@3 满分（相关文档都进了前 3），但常排第 2、第 3，MRR 反而不高 —— 这正是"召回强但排序一般"的典型信号。

## 标注方式的坑

- **别用关键词自评**：拿"答案里有没有出现某关键词"当相关性裁判，会奖励表面词汇重合、惩罚同义改写，评估失真。
- 正确做法是**人工标注 gold 文档**做裁判，关键词命中最多当诊断信息看。
- 评估集要覆盖：词法失配、同义改写、跨主题 hard-negative（长得像但不相关），才能区分策略优劣。
- 小数据集（几十条）结论是**方向性**的，不是统计显著，别过度解读。

## 生成层指标（延伸）

- **忠实度 Faithfulness**：答案是否被检索内容支撑（不编）。
- **答案相关性 Answer Relevance**：答案是否切题。
- **上下文相关性 Context Relevance**：召回的片段是否真有用。
- 工程上常用 RAGAS 等框架，或用更强的模型当"裁判"（LLM-as-a-judge）打分。
